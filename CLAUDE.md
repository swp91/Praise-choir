# Praise Choir Project Guide for Claude Code

이 파일은 Claude Code가 이 저장소에서 작업할 때 따라야 할 프로젝트 규칙이다. 목표는 Codex와 Claude Code가 같은 기준으로 일하게 만드는 것이다.

## 프로젝트 개요

프레이즈찬양대 홈페이지와 관리자 페이지를 만드는 Next.js 프로젝트다.

- 공개 홈페이지: 메인, 대원, 임원, 연습/목표, 일정, 갤러리
- 관리자 페이지: 비밀번호 기반 접근 후 Supabase 데이터를 관리
- 데이터 저장소: Supabase Postgres + Supabase Storage
- 배포 목표: Vercel

관리자 페이지의 핵심 목표는 개발 지식이 없는 사람이 성가대원, 스태프, 임원, 일정, 목표, 사진 등을 직접 관리할 수 있게 하는 것이다.

## 기술 스택

- Framework: Next.js App Router
- Language: TypeScript
- Styling: Tailwind CSS
- Database: Supabase Postgres
- Storage: Supabase Storage
- Admin auth: 단순 비밀번호 + 서버 쿠키 방식
- Package manager: npm

Supabase Auth, 이메일 로그인, 소셜 로그인, OAuth는 현재 관리자 인증 방식에 사용하지 않는다.

## 작업 전 기본 태도

작업을 시작하면 먼저 현재 코드와 DB 구조를 읽고 판단한다. 과거 지식이나 추측으로 바로 수정하지 않는다.

- 파일 검색은 `rg`를 우선 사용한다.
- 기존 패턴을 먼저 따른다.
- 관련 없는 리팩터링은 하지 않는다.
- 사용자가 만든 변경사항을 되돌리지 않는다.
- 작업트리가 더럽다면 내 작업과 무관한 변경은 건드리지 않는다.
- 큰 기능은 설계와 영향 범위를 먼저 설명하고 진행한다.
- 작은 스타일/문구 수정은 과하게 검사하지 않아도 된다.

## 사용자와의 협업 방식

사용자는 한국어로 작업한다. 답변과 커밋 설명은 한국어 중심으로 한다.

- 중간 진행 상황을 짧게 공유한다.
- 막히면 무엇이 막혔는지 구체적으로 말한다.
- 사용자가 명확히 구현을 요청하면 제안만 하지 말고 끝까지 구현한다.
- 새 작업을 시작하기 전에 관련 파일, 최근 커밋, DB 상태를 확인한다.
- 실수 가능성이 있는 DB 변경은 실행 전 의도를 분명히 한다.

## 커밋 규칙

코드나 DB 연동 관련 파일을 변경하면 커밋한다.

커밋 메시지는 Conventional Commit 형식을 사용하되, 타입은 영어, 설명은 한국어로 쓴다.

예시:

- `feat: 임원 관리 페이지 추가`
- `fix: 스태프 섹션을 파트 통계에서 제외`
- `style: 일정 목록 헤더 스타일 통일`
- `test: 일정 폼과 연도 정렬 테스트 갱신`

커밋 전에는 최소한 다음을 확인한다.

- 작업 범위가 작고 스타일만 바꾼 경우: diff 확인 후 커밋
- 기능/DB/서버 액션 변경인 경우: `npx.cmd tsc --noEmit`
- 큰 기능 변경인 경우: `npm.cmd run lint`, 가능하면 `npm.cmd run build`

Google Fonts 때문에 빌드가 네트워크 오류로 실패할 수 있다. 이 경우 코드 오류와 네트워크 오류를 구분해 사용자에게 설명한다.

## 관리자 인증 규칙

관리자 인증은 단순 비밀번호 + 서버 쿠키 방식이다.

- Supabase Auth를 도입하지 않는다.
- 이메일/소셜 로그인으로 바꾸지 않는다.
- 비밀번호 검증은 서버에서만 한다.
- 관리자 Server Action은 매번 쿠키 인증을 확인한다.
- 비밀번호, service role key, secret은 코드나 로그에 남기지 않는다.

## Supabase 작업 규칙

Supabase 작업은 신중하게 한다.

- `service_role` 키는 서버에서만 사용한다.
- 클라이언트 컴포넌트에 관리자 키가 노출되면 안 된다.
- public schema 테이블은 RLS 상태와 정책을 의식한다.
- DDL 변경 후 가능하면 Supabase advisor를 확인한다.
- 테이블 변경은 마이그레이션으로 남긴다.
- 데이터 복구나 대량 업데이트는 실행 전 대상자를 명확히 확인한다.

현재 중요한 테이블:

- `people`: 사람 기본 정보
- `person_private`: 비공개 원본 개인 정보
- `sections`: 파트/악단/스태프 섹션
- `section_memberships`: 사람과 섹션 연결
- `leadership_assignments`: 스태프와 임원 임명
- `media_assets`: Storage/외부 이미지 메타데이터
- `annual_profiles`, `goal_items`: 표어와 목표
- `event_years`, `events`: 일정/보고
- `gallery_albums`, `gallery_items`: 갤러리

## 대원/스태프/임원 데이터 모델

대원 정보와 임원 정보는 분리한다.

- 사람의 기본 정보는 `people`
- 파트 소속은 `section_memberships`
- 상단 지휘/반주/편곡 스태프와 아래 임원진은 `leadership_assignments`

중요한 운영 규칙:

- 지휘/반주/편곡자는 `스태프` 섹션에 넣어 대원관리에서 볼 수 있게 한다.
- `스태프` 섹션은 `sections.kind = 'other'`로 둔다.
- 메인 페이지의 파트 수 통계는 `kind != 'other'`만 센다.
- 따라서 스태프 섹션이 있어도 메인 통계의 파트 수는 6개로 유지된다.
- 아래 임원진은 기존 대원 중에서 임명한다.
- 임원을 삭제해도 대원 정보는 삭제하지 않는다.

## 관리자 UI 원칙

관리자 페이지는 “개발 지식이 없는 사용자”가 쓰는 운영 도구다.

- 기본 화면은 읽기 쉬운 텍스트 보기여야 한다.
- 수정 버튼을 눌렀을 때만 폼으로 전환한다.
- 삭제는 반드시 확인 모달을 띄운다.
- 공개/비공개는 대원관리와 같은 토글 버튼 스타일을 쓴다.
- 정렬이 필요한 목록은 가능하면 드래그앤드롭으로 제공한다.
- 수정/저장/취소 후 화면이 맨 위로 튀지 않게 앵커를 붙인다.
- 테이블 헤더와 버튼 컬럼 정렬은 대원관리 스타일을 기준으로 맞춘다.

## 현재 구현 상태 요약

완료된 큰 흐름:

- Supabase 연결
- 대원관리 CRUD와 사진 업로드
- Cloudinary 이미지의 Supabase Storage 이관
- 표어/목표 관리
- 일정관리
- 임원관리
- 스태프 섹션 추가

주의할 점:

- `role_en`은 제거되었다. 영문 직무 항목을 다시 만들지 않는다.
- 스태프 공개/비공개 항목은 제거되었다.
- 스태프 삭제는 확인 모달이 필요하다.
- 임원 삭제 모달에는 “임원 명단에서만 삭제되고 대원 정보는 남는다”는 안내가 있어야 한다.

## 검증 습관

작업 후 성공했다고 말하기 전에 실제 확인 결과를 말한다.

권장 검증:

- `npx.cmd tsc --noEmit`
- `npm.cmd run lint`
- `npm.cmd run build`
- Supabase MCP SQL 조회
- Supabase security advisor

작은 스타일 변경은 사용자가 원한 대로 타입/빌드까지 매번 돌리지 않아도 된다. 하지만 기능, DB, 서버 액션, 데이터 마이그레이션은 검증한다.

## 답변 스타일

- 한국어로 간결하게 말한다.
- 무엇을 바꿨는지, 무엇을 확인했는지, 커밋 해시를 알려준다.
- 불확실한 것은 불확실하다고 말하고 확인한다.
- 사용자가 질문한 것보다 더 큰 위험이 보이면 짧게 짚어준다.
- 장황한 설명보다 현재 작업에 필요한 핵심을 우선한다.
