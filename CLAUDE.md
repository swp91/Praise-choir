# 성가대 홈페이지 프로젝트

성가대(찬양대)를 위한 커뮤니티 홈페이지. 단원 관리와 공지·일정 전달을 중심으로 시작해, 추후 연습용 음원/영상 공유와 출석 관리 기능으로 확장될 예정이다.

## 현재 상태

- **공개 홈페이지**: 대체로 완성. 공지·일정 등 정보 전달 기능 동작.
- **데이터/스토리지**: Supabase DB 및 Storage 연결 완료.
- **관리자 페이지**: 진행 중 (약 40%). 단원 관리, 공지/일정 작성·수정 기능 구축 중.
- **향후 확장(아직 미착수)**: 연습용 음원/영상 업로드 및 공유, 유튜브 영상 임베드, 출석 체크 프로그램.

새 작업을 시작하기 전에 위 상태를 기준으로 현재 코드베이스를 먼저 확인하고, 추측으로 이미 있는 기능을 다시 만들지 않는다.

## 기술 스택

- **프레임워크**: Next.js (App Router)
- **언어**: TypeScript (strict)
- **데이터/스토리지**: Supabase (Postgres, Storage)
- **관리자 인증**: 단순 비밀번호 인증 + 서버 측 쿠키 방식. Supabase Auth, 이메일 로그인, 소셜 로그인을 **사용하지 않는다.**
- **배포**: Vercel
- **스타일링**: 기존 코드베이스의 컨벤션을 따른다(임의로 새 스타일링 라이브러리를 도입하지 않는다).

## 작업 방식 (중요)

코드를 바로 작성하기 전에, 작업 성격에 따라 아래 흐름을 따른다.

- **새 기능 구현이나 구조 설계**(예: 출석 기능 추가, 음원 공유 모듈 설계, 관리자 페이지의 새 섹션)를 시작할 때는 곧장 코드부터 짜지 말고, 먼저 superpowers의 **brainstorm** 스킬로 "진짜 무엇을 만들려는지"와 요구사항·제약을 명확히 한다. 합의된 설계가 나오면 **write-plan**으로 단계별 구현 계획을 세우고, 승인 후 **execute-plan**으로 실행한다.
- **단순 수정**(버그 픽스, 문구 변경, 스타일 조정, 한두 줄 변경)은 브레인스토밍 단계 없이 바로 처리한다. 불필요하게 "무엇을 하려느냐"를 되묻지 않는다.
- 요구사항이 모호하면 추측해서 진행하지 말고 먼저 짧게 질문한다. 단, 한 번에 핵심 질문 하나만.

## 코드 컨벤션

- **주석과 커밋 메시지는 한국어로 작성한다.**
- 기존 파일의 패턴(폴더 구조, 네이밍, import 정렬, 컴포넌트 작성 방식)을 먼저 읽고 그에 맞춘다. 자기 취향대로 새 패턴을 도입하지 않는다.
- TypeScript 타입을 우선한다. `any`는 피하고, 불가피하면 이유를 주석으로 남긴다.
- 서버 컴포넌트와 클라이언트 컴포넌트의 경계를 명확히 한다. 클라이언트 전용 훅/상태가 필요한 곳에만 `"use client"`를 붙인다.
- 새 의존성(npm 패키지)을 추가할 때는 먼저 알리고 이유를 설명한다. 임의로 설치하지 않는다.

## 관리자 인증 (중요)

관리자 페이지는 **단순 비밀번호 인증 + 서버 측 쿠키** 방식을 쓴다. 일부 임원이 공유받은 단일 비밀번호를 입력해 들어오는 구조다.

- **Supabase Auth, 이메일 로그인, 소셜 로그인, OAuth, 매직링크 등을 제안하거나 도입하지 않는다.** 인증 작업을 할 때 임의로 이런 방식으로 갈아엎으려 하지 말 것. 현재 구조를 유지·보완하는 방향으로만 작업한다.
- 비밀번호 검증과 세션 판단은 **서버 측에서** 이루어져야 한다. 비밀번호를 클라이언트 코드에 노출하거나 클라이언트에서 비교하지 않는다.
- 비밀번호 원문은 환경 변수로만 다루고 커밋·로그·클라이언트 번들에 절대 남기지 않는다. 가능하면 평문 비교 대신 해시 비교를 권장하되, 구조 변경이 필요하면 먼저 알린다.
- 인증 쿠키는 `httpOnly`, `secure`, 적절한 `sameSite`, 만료 시간을 설정한다. 관리자 전용 라우트/Server Action은 매 요청마다 쿠키를 검증한다.
- 보안상 더 나은 방식(예: 임원별 개별 비밀번호, 해싱 강화 등)이 떠오르면 강요하지 말고 선택지로만 짧게 언급한다. 기본은 현재 방식을 존중한다.

## Supabase (DB / Storage) 주의사항

- DB 스키마를 변경하거나 마이그레이션이 필요한 작업은 실행 전에 반드시 먼저 알리고, 변경 내용(테이블/컬럼/정책)을 요약해 보여준 뒤 진행한다.
- **RLS(Row Level Security) 정책**을 항상 의식한다. 단원 데이터나 관리자 전용 데이터는 정책 없이 공개하지 않는다. 다만 관리자 인증이 Supabase Auth 기반이 아니므로, RLS만으로 관리자 경계를 가정하지 말고 서버 측 쿠키 검증과 함께 권한을 판단한다.
- Storage(음원·영상·이미지) 접근도 버킷 정책으로 보호한다. 누구나 봐도 되는 정적 자산과 단원/관리자 전용 자료를 구분해 다룬다.
- service role 키 등 민감한 키는 절대 클라이언트 코드나 커밋에 노출하지 않는다. 환경 변수(`.env`)로만 다루고, 노출 위험이 보이면 즉시 경고한다.

## 보안·개인정보

- 단원의 개인정보(이름, 연락처 등)를 다루는 화면과 API는 인증·인가를 반드시 거친다.
- 비밀키, 토큰, 비밀번호를 코드·로그·커밋에 남기지 않는다.
- 사용자 입력은 신뢰하지 않는다. 폼 제출, 파일 업로드 등은 서버 측 검증을 거친다.

## 확장 기능 설계 시 고려사항

향후 음원/영상 공유와 출석 프로그램을 붙일 때를 대비해, 지금 만드는 구조가 나중에 확장 가능하도록 과도하게 특정 가정에 묶지 않는다. 다만 아직 없는 기능을 미리 구현하지는 않는다(YAGNI). 확장 가능성은 폴더/모듈 경계를 깔끔히 두는 선에서만 고려한다.

## 응답 방식

- 한국어로, 직접적이고 솔직하게 답한다. 불필요한 사족이나 과한 칭찬은 생략한다.
- 코드를 수정하면 무엇을 왜 바꿨는지 핵심만 요약한다.
- 잘못된 접근이라고 판단되면 그대로 따르지 말고 근거와 함께 대안을 제시한다.
